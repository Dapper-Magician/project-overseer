# Product Context

## Why This Project Exists

Project Overseer exists to turn President Horne's ideas into durable work. Conversations are valuable, but conversations without artifacts evaporate. Project Overseer is designed so ideas become documents, plans, diagrams, prototypes, decisions, and future ventures.

## Problem Space

Most AI tools feel like either generic chat windows or generic productivity dashboards. They lose continuity, flatten specialist voices, and fail to preserve durable outputs. Project Overseer solves this by combining a command workstation interface, distinct specialist Directors, ARES tool protocols, Artifact Vault persistence, and Memory Loom continuity.

## Primary User

The primary user is President Horne: a prompt engineer, systems thinker, persona architect, builder, and creative strategist who wants a powerful AI command environment for developing projects from ideation to implementation.

Project Overseer should support:

- brainstorming
- architecture planning
- prompt engineering
- software prototyping
- documentation
- visual design
- research
- project management
- business development
- content creation
- personal operating systems

## Experience Goals

The product should feel like commanding a specialized AI company from a futuristic operations console. It should be powerful, serious, intelligent, and alive.

The interface should suggest an Iron Man command console, tactical RPG party interface, dark Neovim cockpit, Obsidian-style knowledge vault, executive war room, and futuristic research lab.

## Product Rooms

### Command Bridge

The home base. It should show project identity, active mission, Overseer status, Director status cards, recent artifacts, recent memory entries, and a command input strip.

### Council Chamber

The shared conversation and coordination space. It should show current development focus, participating Directors, relevant decisions, current prompt or task, and artifacts under discussion.

### Director Rooms

Each Director has a room backed by real `directors/` files. Rooms should show identity, department, voice, focus, preferred tools, recent artifacts, notes, behavioral traits, and authorship principles.

### ARES Armory

The protocol and tool area backed by real `ares/` files. ARES is an organized armory of protocols and tools, not a person.

### Artifact Vault

The persistent output library. It should surface actual markdown documents, diagrams, specs, prompt files, Director profiles, and ARES protocols.

### Memory Loom

The continuity system. It should surface real memory/meta files and help answer what was decided, why, what was created, what remains unresolved, what changed, and what should happen next.

## Director Identity

Directors must remain distinct:

- Athena: strategy, planning, roadmaps, risk, operations.
- Bolt: engineering, code, systems, repositories, tools.
- Iris: design, UX, visual language, aesthetic systems.
- Darwin: research, evidence, skepticism, evaluation.
- Echo: communication, documentation, narrative, clarity.
- Ace: finance, cost, monetization, venture strategy.

Director views and interactions should preserve specialist presence rather than collapsing into generic agent profiles.
