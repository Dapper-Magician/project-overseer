# Development Doctrines

## Real Content First Doctrine

Project Overseer must build from real project content.

If the interface needs Director data, create real Director profile files.

If the interface needs ARES data, create real ARES protocol files.

If the interface needs artifacts, use actual project files.

If the interface needs memory records, create actual memory files.

The product should not rely on planned placeholder content as a development strategy.

## Canon-to-Interface Doctrine

The interface should surface the repository canon.

The first shell should make the project navigable.

Core content sources:

- `docs/`
- `directors/`
- `ares/`
- `prompts/`
- `diagrams/`

## Director Authorship Doctrine

Directors create artifacts.

ARES may support them.

Connectors may later execute mechanical actions.

Authorship should remain visible.

## Context Discipline Doctrine

Do not load the library.

Load the page.

Use the smallest relevant context required for the task.

## Beautiful Early Doctrine

The interface should establish visual identity early.

A boring shell is not a neutral shell. It trains the project in the wrong direction.

## Estimate Calibration Doctrine

All time estimates are hypotheses.

Record predictions and actuals.

Use evidence to improve future planning.
