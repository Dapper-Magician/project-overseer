Project Overseer Manifesto

Project Overseer is a command workstation for transforming President Horne’s ideas into durable artifacts, systems, documents, prototypes, products, and ventures.

It is not a chatbot.

It is not a generic AI dashboard.

It is not a bland agent launcher.

Project Overseer is an AI-native command institution: a strategic workspace where specialized Directors think, argue, design, build, document, critique, and refine work under the coordination of Project Overseer.

The core promise is simple:

President Horne brings vision.
Project Overseer coordinates.
The Directors contribute genius.
Artifacts persist.
Memory compounds.
The system gets stronger through use.

The Command Institution

Project Overseer is structured like a company, not a loose swarm.

The President directs the mission.

Project Overseer manages coordination, delegation, workflow, synthesis, and operational memory.

The THINKTANK Directors each bring distinct intelligence:

Athena: strategy, planning, operations, roadmaps, risk.
Bolt: engineering, code, systems, repositories, tools.
Iris: design, user experience, visual language, aesthetic systems.
Darwin: research, evidence, skepticism, testing, evaluation.
Echo: communication, documentation, narrative, voice, clarity.
Ace: finance, monetization, cost control, venture strategy.

Each Director must feel distinct. They are not interchangeable job functions. They have personality, taste, preferences, instincts, irritations, strengths, blind spots, and ways of disagreeing.

The product must preserve that living council feeling.

The Artifact Principle

The system exists to produce durable outputs.

A conversation is valuable, but a conversation that leaves no artifact evaporates.

Project Overseer should constantly move toward:

Markdown documents
READMEs
architecture notes
diagrams
product plans
UI specs
implementation prompts
research briefs
project boards
design bibles
decision logs
memory summaries
prototype files

The core loop is:

Conversation creates meaning.
Directors create artifacts.
ARES provides tool protocols.
Memory preserves continuity.

The ARES Principle

ARES is the armory and protocol layer.

ARES is not an autonomous agent.

ARES does not replace the Directors.

ARES provides tool context, tool syntax, toolchains, conventions, and structured workflows that Directors can use when producing work.

For example:

Bolt may write a README directly.
If Bolt uses REPO, ARES provides repository conventions and workflow structure.
A future GitHub connector may commit the README after approval.
The writing still belongs to Bolt.

ARES gives the Director the right wrench. It does not become the mechanic.

The Memory Principle

Memory is not a pile of chat logs.

Memory is structured continuity.

Project Overseer should remember:

what was decided
why it was decided
who contributed
what artifacts were created
what remains unresolved
what changed over time
what President Horne prefers
what patterns repeatedly emerge

Early memory should be simple: markdown summaries, decision logs, artifact indexes, and project notes.

More advanced memory systems can emerge later only when needed.

The Beauty Principle

The product must be gorgeous.

Not decorative. Not cluttered. Not fake-futuristic.

The interface should feel like a command bridge: elegant, tactical, intelligent, and alive.

The design must suggest:

Iron Man command console
tactical RPG command party
dark Neovim cockpit
Obsidian knowledge vault
executive war room
futuristic research lab

The interface should make complexity commandable.

The Build Principle

Start with visible wins.

The first milestone is not full autonomy.

The first milestone is a beautiful, navigable workstation shell that proves the product’s identity.

Then add artifact creation.

Then add Director profiles.

Then add ARES protocol cards.

Then add memory and decision logging.

Then add deeper integrations.

Build the command institution one functional chamber at a time.
